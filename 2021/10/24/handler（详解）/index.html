handler（详解）

Handler是什么？

Handler 是 Android 中的异步消息处理机制，使用者可以在不阻塞 UI 线程的前提下轻松的实现消息管理和发送。

handler机制中包含4个关键类： Message（消息），MessageQueue（消息队列），Looper(轮询器)，Handler（消息发送和接收并处理）。

handler 负责发送 message，将其加入到 MessageQueue 中，Looper 不间断的从 MessageQueue 中取出消息，并发送给对应的handler 实例去处理

Handler 的使用

  • 创建 handler 对象（现在 new Handler() 方法已经被注释为 @Deprecated 了，改用为 new Handler(Looper.getMainLooper()) 来创建主线程的 handler 了），并重写了 handleMessage() 方法

  • 在需要的地方，创建 Message 对象，根据需要可以在 Message 对象中携带一些数据，调用 handler.sendMessage(msg) 方法即可将 Message 发送到 MessageQueue 中，之后便被取出并在 handleMessgae 方法中针对具体的 Message 对象进行相应的处理。

    message 可以存储的信息：

    1. message.what: 用户定义的消息代码，以便收件人可以识别
    2. message.arg1 定义传递的 int 类型的值1
    3. message.arg2 定义传递的 int 类型的值2
    4. message.obj 定义传递的值由于类型是 object(对象) 所以比较常用,可以传递各种值
    5. handler.obtainMessage() 带参数形式发送消息,主要的却别代价可以通过下面的 demo 区分,
    6. message.setData() 使用 bundle 的实行传参

ThreadLoacl

ThreadLocal使用及其实现原理

[Java 之 ThreadLocal 详解](Java 之 ThreadLocal 详解 - 掘金 (juejin.cn))

ThreadLocal 是一个将在多线程中为每一个线程创建单独的变量副本的类; 当使用 ThreadLocal 来维护变量时, ThreadLocal 会为每个线程创建单独的变量副本, 即每个线程内部都会有一个该变量，且在线程内部任何地方都可以使用，线程之间互不影响, 避免因多线程操作共享变量而导致的数据不一致的情况。

ThreadLocal 的作用和同步机制有些相反：同步机制是为了保证多线程环境下数据的一致性；而 ThreadLocal 是保证了多线程环境下数据的独立性。

ThreadLocal 的构造函数只有一个简单的无参构造函数，并且没有任何实现。下面以 set 和 get 方法分析，大致了解下 ThreadLocal 工作原理

ThreadLoacl 的使用
set(T value) 方法

set(T value) 方法中，首先获取当前线程，然后在获取到当前线程的 ThreadLocalMap，如果 ThreadLocalMap 不为 null，则将 value 保存到 ThreadLocalMap 中，并用当前 ThreadLocal 作为 key；否则创建一个 ThreadLocalMap 并给到当前线程，然后保存 value。

get() 方法

同样的，在 get() 方法中也会获取到当前线程的 ThreadLocalMap，如果 ThreadLocalMap 不为 null，则获取 key 为当前 ThreadLocal 的 value 的值；否则调用 setInitialValue() 方法返回初始值，并保存到新创建的 ThreadLocalMap 中。

ThreadLocalMap

ThreadLocalMap 相当于一个 “HashMap”，是真正保存值的地方。

  • ThreadLocalMap 以 ThreadLocal 为 key。一个线程可以 new 多个 ThreadLocal 对象进行存取数据
  • ThreadLocalMap 是一个 static class 类，它的引用在Thread类里。因此 当前线程，因此不存在多线程操作，因此未进行加锁以及同步操作。
存储结构

ThreadLocalMap 内部维护了一个哈希表（ Entry[] 数组）来存储数据，并且定义了加载因子为 2 / 3；同时采用线性探测法解决冲突

Entry 是 ThreadLocalMap 中的一个静态内部类

  • Entry 用于保存一个键值对，其中 key 以弱引用的方式保存
内存泄漏

在 ThreadLocalMap 的 set()，get() 和 remove() 方法中，都有清除无效 Entry 的操作，这样做是为了降低内存泄漏发生的可能。

Entry 中的 key 使用了弱引用的方式，这样做是为了降低内存泄漏发生的概率，但不能完全避免内存泄漏。

这句话的意思好象是矛盾的，下面来分析一下。

​ 在垃圾回收器线程扫描它所管辖的内存区域的过程中，一旦发现了只具有弱引用的对象，不管当前内存空间足够与否，都会回收它的内存。不过，由于垃圾回收器是一个优先级很低的线程，因此不一定会很快发现那些只具有弱引用的对象。

假设 Entry 的 key 没有使用弱引用的方式，而是使用了强引用：由于 ThreadLocalMap 的生命周期和当前线程一样长，

  • 那么当引用 ThreadLocal 的对象被回收后，由于 ThreadLocalMap 还持有 ThreadLocal 和对应 value 的强引用，ThreadLocal 和对应的 value 是不会被回收的，这就导致了内存泄漏。
  • 那么当持有value的强引用释放掉后，当线程没有回收释放时，threadLocalMap 会一直持有 ThreadLocal 以及 value 的强应用，导致 value 不能够被回收，从而造成内存泄漏。

所以 Entry 以弱引用的方式避免了 ThreadLocal 没有被回收而导致的内存泄漏，当ThreadLocal的强引用释放掉后，通过一次系统gc检查，发现ThreadLocal对象只有threadLocalMap中Entry的弱引用持有，此时根据弱引用的机制就会回收 ThreadLocal 对象，从而避免了内存泄露。

但是此时第一种情况下 value 仍然是无法回收的，依然会导致内存泄漏：

  • ThreadLocalMap 已经考虑到这种情况，并且有一些防护措施：在调用 ThreadLocal 的 get()，set() 和 remove() 的时候都会清除当前线程 ThreadLocalMap 中所有 key 为 null 的 value。这样可以降低内存泄漏发生的概率。所以我们在使用 ThreadLocal 的时候，每次用完 ThreadLocal 都调用 remove() 方法，清除数据，防止内存泄漏。

如何正确使用同步屏障/异步消息

异步消息需要同步屏障的辅助，但同步屏障我们无法手动添加，因此了解系统何时添加和删除同步屏障是非常必要的。只有这样，才能更好地运用异步消息这个功能，知道为什么要用和如何用。

了解同步屏障需要简单了解一点屏幕刷新机制的内容。放心，只需要了解一丢丢就可以了。

我们的手机屏幕刷新频率有不同的类型，60Hz、120Hz等。60Hz表示屏幕在一秒内刷新60次，也就是每隔16.6ms刷新一次。屏幕会在每次刷新的时候发出一个 VSYNC 信号，通知CPU进行绘制计算。具体到我们的代码中，可以认为就是执行onMesure()、onLayout()、onDraw()这些方法。好了，大概了解这么多就可以了。

了解过 view 绘制原理的读者应该知道，view 绘制的起点是在 viewRootImpl.requestLayout() 方法开始，这个方法会去执行上面的三大绘制任务，就是测量布局绘制。但是，重点来了：

调用requestLayout()方法之后，并不会马上开始进行绘制任务，而是会给主线程设置一个同步屏障，并设置 VSYNC 信号监听。

（invalidate() 方法和 requestLayout() 处理逻辑基本相同）
当 VSYNC 信号的到来，VSYNC 信号监听会监听到 VSYNC 信号的到来并发送一个异步消息到主线程Handler，执行我们上一步设置的绘制监听任务，并移除同步屏障

这里我们只需要明确一个情况：调用 requestLayout() 方法之后会设置一个同步屏障，直到 VSYNC 信号到来才会执行绘制任务并移除同步屏障。 requestLayout() 方法只有在有绘制任务的时候，才会被调用。

同步屏障在队列中，只能存在一个，有标志位记录，当第一次加入同步屏障时，标志位会记录已经存在同步屏障了，当同步屏障被移除时，标志位会被还原

设置同步屏障的函数中，还会设置 VSYNC 信号监听。因此，如果没有设置同步屏障，也不会存在 VSYNC 信号监听，因为也不会接受处理 VSYNC 信号并发送异步消息

同步屏障是放在队列头的，这样next方法才可以获取到屏障忽略后续的同步 message，优先处理异步 message

我们应该注意的一点是：

不可在主线程执行重量级任务，无论异步还是同步。

那，我们以后岂不是可以直接使用异步Handler来取代同步Handler了？是，也不是。

同步Handler有一个特点是会遵循与绘制任务的顺序，设置同步屏障之后，会等待绘制任务完成，才会执行同步任务；而异步任务与绘制任务的先后顺序无法保证，在等待ASYNC的期间可能被执行，也有可能在绘制完成之后执行。因此，我的建议是：如果需要保证与绘制任务的顺序，使用同步Handler；其他，使用异步Handler。

Handler 内存泄漏

Handler造成内存泄露的原因。非静态内部类，或者匿名内部类。使得Handler默认持有外部类的引用。在Activity销毁时，由于Handler可能有未执行完/正在执行的Message。导致Handler 持有 Activity 的引用。进而导致 GC 无法回收 Activity。

handler.removeCallbacksAndMessages(null)

Activity 销毁时，调用 mHandler.removeCallbacksAndMessages(null) 清空 Handler 中未执行或正在执行的Callback以及Message

mHandler.removeCallbacksAndMessages() 方法参数传入 null，表示会将所有的Callbacks和Messages全部清除掉

我们来分析下 mHandler.removeCallbacksAndMessages(null) 方法。

该方法中调用了 MessageQueue 的 removeCallbacksAndMessages() 方法，我们继续来看 MessageQueue 的 removeCallbacksAndMessages() 方法。

  • mMessages 保存的是链表头部

  • 第一个 while() 循环从链表头部开始，往后依次遍历，找到第一个不满足删除条件的结点即停止。遍历过程中，满足删除条件的结点，调用 recycleUnchecked() 方法移除。

    该 while() 循环目的为找到一个不满足删除条件的结点，并将其更新成 mMessages 的值，即成为新的链表头部

  • 第二个 while() 循环即遍历到链表尾部，在遍历过程中，如果结点满足删除条件，则调用 recycleUnchecked() 方法移除

至此，链表中所有满足删除条件的结点即被删除。

HandlerThread 的使用范围很有限，一般都是需要在子线程不断接受消息处理耗时任务，如在子线程不断地获取数据更新UI的时候，就可以用到handlerThread。HandlerThread 即同时帮你做了两件事：

  1. 创建了一个线程
  2. 开启了该子线程的 looper，并创建了该线程的 handler

HandlerThread 原理

HandlerThread 继承自 Thread，整个类的代码不是很多，重点在run()和getLooper()方法：

  • getLooper()方法：

    如果 Looper 还未初始化完成，则调用 wait() 进行等待，等到 Looper 初始化完成了之后就会唤醒他，然后成功返回 Looper，这样就不会出现 Looper 还未初始化完成就被使用了的情况了。

  • run() 方法：

    • 当调用 HandlerThread 的 start() 方法后，会自动调用到 run() 方法
    • 调用了 Looper.prepare() 初始化 looper
    • 当 Looper.prepare() 执行完，即 looper 初始化完成后，将 mLooper 赋值为 looper 并调用 notifyAll() 唤醒等待。（即上方的 getLooper() 如果等待了 即会被唤醒）

View.post()

View.post()的使用

  • View.post() 中可以对 UI 进行操作，即使是在子线程中调用 View.post()
  • View.post() 中的操作执行时，View 的宽高已经计算完毕了

View.post() 的原理

Android：阿里P7大牛教你区分Handler.post和View.post的区别

看方法可大概知道：

  • 如果 mAttachInfo 不为 null，则将任务交由attachInfo 中的 Handler 处理

  • 如果 mAttachInfo 为 null，则调用 getRunQueue().post(action)

    从 mRunQueue 中的注释可以看出，mRunQueue 就是 View 用来处理它还没 attach 到 window 时，客户代码发起的 post 调用的

总结

  1. Handler.post，它的执行时间基本是等同于onCreate里那行代码触达的时间；

  2. View.post，则不同，它说白了执行时间一定是在 Act#onResume 发生后才开始算的（在ViewRootImpl.performTraversals()中触发的）

    onMesasure()，onLayout()，onDraw() 也在 ViewRootImpl.performTraversals()中触发，不过 View.post 在他们之后触发，因此 view.post 有时也用来获得控件的大小。

  3. 当然，虽然这里说的是post方法，但对应的postDelayed方法区别也是类似的。

平时当你项目很小，MainActivity的逻辑也很简单时是看不出啥区别的，但当act的onCreate到onResume之间耗时比较久时（比如2s以上），就能明显感受到这2者的区别了，而且本身它们的实际含义也是很不同的，前者的Runnable真正执行时，可能act的整个view层次都还没完整的measure、layout完成，但后者的Runnable执行时，则一定能保证act的view层次结构已经measure、layout并且至少绘制完成了一次。
